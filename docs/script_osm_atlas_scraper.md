# Script Architecture: `osm_atlas_scraper.mjs`

## Executive Summary
This script is a lightweight, rapid-prototyping REST client built to extract micro-sets of geospatial railway data directly from the Overpass API framework. It serves as the alternative to downloading a massive 1.6GB PBF file, enabling developers to scrape bounding box sectors (like a specific city limits) in plain JSON for immediate front-end testing.

## Architecture & Logic Flow

The flow is built exclusively on standard Node.js (`fetch`, `fs`) avoiding all third party overhead dependencies.

**Logic Flow:**
1. **Query Formulation:** An Overpass Query Language (QL) block is defined requesting `railway=rail` and `railway=station` targeting explicitly within a dynamic `bbox` (Bounding Box Coordinate Matrix, e.g. `28.5,77.1,28.7,77.3`). 
2. **REST Execution:** Fire an HTTP `POST` against `https://overpass-api.de/api/interpreter`. The query payload asks the Overpass server to dynamically filter and calculate the geometry payload.
3. **Data Localization:** Extracts the returned `JSON` body and dynamically saves it to `.tmp/osm_railways_{bbox}.json` without any transformation.

## Extensibility & Usage Boundaries 

* **The Size Limit:** This script is hard-capped conceptually by Overpass timeouts `[timeout:25];` and physical bandwidth quotas. Attempting to pass the bounding box of the entire Indian subcontinent will result in `HTTP 429 Too Many Requests` or outright Server aborts. This is explicitly meant for city-wide or regional testing, while the `import_pbf_atlas.ts` handles the macro ecosystem.
* **Integration Points:** The `.tmp/` output JSON files generated by this script correspond uniquely to the `import_osm_atlas.ts` pipeline injector as opposed to the PBF variants.

```javascript
// Example modification for targeting Mumbai instead of Delhi
const mumbaiBBox = '18.8,72.7,19.3,73.1';
fetchOSMRailways(mumbaiBBox).catch(console.error);
```
